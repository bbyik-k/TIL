# [Core JS] 1-6. undefined와 null

## 요약

- undefined는 자바스크립트 엔진이 값이 없음을 나타내기 위해 자동으로 부여하는 값이고, null은 개발자가 의도적으로 “없음”을 표현하기 위해 사용하는 값.
- 두 값은 비슷해 보이지만 생성 주체와 의미가 다르며, 이를 구분하지 못하면 디버깅과 상태 관리에서 혼란이 발생.

## 핵심 내용

- undefined와 null은 모두 “값이 없음”을 표현하지만, 발생 원인과 역할이 다르다.
- 특히 배열, 변수 선언, 비교 연산에서 이 차이가 실제 동작 차이로 드러난다.

### 핵심내용 1

**undefined는 자바스크립트 엔진이 자동으로 부여하는 값이다.**  
변수를 선언만 하고 값을 대입하지 않았을 때, 존재하지 않는 프로퍼티에 접근했을 때, return 값이 없는 함수의 실행 결과로 엔진이 undefined를 반환한다. 이는 “값을 아직 할당받지 못한 상태”를 의미한다.

### 핵심내용 2

**null은 개발자가 명시적으로 ‘값이 없음’을 표현하기 위한 값이다.**  
null은 처음부터 의도적으로 비어 있음을 나타내기 위해 만들어진 데이터 타입이다. 따라서 undefined와 달리, null은 개발자의 의도가 코드에 드러난다.

### 핵심내용 3

**배열에서 undefined와 ‘비어 있음(empty)’은 다르게 동작한다.**  
배열에 undefined를 직접 할당한 요소는 실제 프로퍼티로 존재하며 순회 대상이 된다. 반면, 단순히 length만 늘린 배열의 빈 요소는 존재하지 않는 프로퍼티이기 때문에 forEach, map 등 순회 메서드의 대상에서 제외된다.

## Check Point

- undefined는 “아직 값이 정해지지 않음”
- null은 “의도적으로 비워 둠”
- 배열에서 empty slot과 undefined는 전혀 다르게 동작한다.
- null의 typeof는 object이며, 이는 자바스크립트의 오래된 설계 오류.

## 언제 문제가 되는가 / 언제 쓰는가

- **React 상태 관리**에서 초기값을 undefined로 둘 경우, 의도치 않은 렌더링 분기 오류가 발생할 수 있다.
- API 응답에서 “값이 없음”을 표현할 때 undefined 대신 null을 사용해야 의미가 명확해진다.
- 배열을 map, filter, reduce로 다룰 때 empty slot과 undefined를 혼동하면 예상과 다른 결과가 나온다.

## 정리

- undefined는 엔진의 영역, null은 개발자의 영역이다.
